# Implementation Spec: Inline Research References - Phase 1

**PRD**: ./prd-phase-1.md
**Estimated Effort**: L (Large)

## Technical Approach

The feature spans three layers: (1) Backend prompt modification to produce `{{ref:N}}` markers during existing content writing, (2) Backend post-processing to convert markers into semantic HTML with `artifact_research` ID references, (3) Frontend TipTap custom node + Radix HoverCard for rendering and interaction.

The critical design decision is **zero additional cost**: markers are generated by the same LLM call that writes content. The content-writing prompt already receives research context with numbered indices — we add a single instruction for the LLM to append `{{ref:N}}` after sentences it drew from. Post-processing parses these markers and converts them to `<span>` elements with data attributes linking to the actual `artifact_research` UUIDs.

The humanizer preserves markers by treating `{{ref:N}}` as structural tokens (like `[IMAGE:...]` placeholders) that must not be removed or interpreted as AI patterns.

## UX/UI Design

### Design Intent

**Who is this human?** A 40-something fractional CMO reviewing a blog post that NextUp's AI just generated from 15 research sources. They're scanning the content before publishing, curious about which claims have research backing but not wanting the indicators to interfere with their editing flow.

**What must they accomplish?** Glance at where research influenced the content. Optionally hover to inspect a source. Then continue editing.

**What should this feel like?** Like margin annotations in a well-edited manuscript — present but subordinate. The indicators should feel like a quiet layer of intellectual depth sitting below the surface of the text, visible to those who look but never interrupting those who don't.

### Reference Indicator Design

The indicator is a **small superscript dot-number** — not a bracket, not a pill, not a colored badge. It sits at superscript height after the sentence's terminal punctuation, using the brand's muted foreground color in its default state.

```
Visual Spec:
┌─────────────────────────────────────────────────────┐
│                                                     │
│  Startups save 60-70% compared to full-time hires.¹ │
│                                                  ↑  │
│                                        ref indicator│
│                                                     │
│  Multiple refs on same sentence.¹ ²                 │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**Visual properties:**
- Font size: `10px` (`text-[10px]`)
- Position: `vertical-align: super` with slight `top: -0.3em` offset
- Color (default): `text-muted-foreground` — blends with metadata text
- Color (hover): `text-primary` — cyan glow on hover signals interactivity
- Font weight: `font-medium` (500)
- Cursor: `cursor-pointer` on hover
- Transition: `transition-colors duration-150`
- Margin: `ml-0.5` from preceding text, `mr-0.5` gap between sequential indicators
- Line height: does NOT affect parent line height (critical — `line-height: 0` on the indicator span)
- Min-width: none — indicator is naturally sized to its digit(s)

**States:**
| State | Appearance |
|-------|-----------|
| Default | `text-muted-foreground` — quiet, blends with body text |
| Hover | `text-primary` — cyan (dark mode) / blue (light mode), signals clickability |
| Active/Focused | `text-primary ring-1 ring-primary/30 rounded-sm` — subtle focus ring for accessibility |
| Edit mode | Same as above + cursor shows as pointer |
| Read-only mode | Same visual, not deletable |

**Why this design, not alternatives:**
- **Not brackets `[1]`**: Brackets add visual noise and feel academic — wrong tone for advisory professionals
- **Not colored pills**: Pills (Perplexity-style) are too prominent for content meant to be published; they compete with the text
- **Not underlined text**: Underlines suggest hyperlinks and would confuse the interaction model
- **Superscript number**: Universally understood as a reference, minimal visual weight, doesn't disrupt reading flow. The muted color ensures it recedes until the user deliberately looks for it.

**Interaction model:**
- **Desktop**: Hover indicator → 200ms delay → card appears above (preferred) or below (if no space above)
- **Grace period**: 100ms delay before dismissing when mouse leaves indicator — allows mouse to move into card
- **Card is hoverable**: Mouse inside card keeps it open
- **Dismiss**: Mouse leaves both indicator and card → 100ms → card closes
- **Mobile**: Tap indicator → card appears → tap outside dismisses
- **Keyboard**: Focus indicator with Tab → Enter/Space opens card → Escape closes

**Edge cases:**
- **Missing URL**: "Open Source" button hidden entirely, card is shorter
- **Long title**: `line-clamp-2` with ellipsis
- **Long excerpt**: First sentence only, `line-clamp-2` with ellipsis
- **Card near viewport edge**: Radix handles repositioning (side="top" with fallback to "bottom")

### Multiple References Design

```
Sequential indicators with 2px gap:

  "Startups save 60%.¹ ² This means..."
                     ↑ ↑
                 2px gap between
```

Each indicator is an independent node. No grouping container — TipTap renders them as adjacent inline nodes with `mr-0.5` spacing. Each has its own hover card.

## Implementation Details

### 1. Reference Marker Utilities (`referenceMarkerUtils.ts`)

**Overview**: Pure functions for parsing `{{ref:N}}` markers from LLM output and converting to HTML.

```typescript
interface ResearchItem {
  id: string;          // artifact_research UUID
  index: number;       // 1-based position in the research context
  source_name: string;
  source_type: string;
  source_url: string | null;
  excerpt: string;
}

/**
 * Parse {{ref:N}} markers from content and convert to HTML spans.
 * Returns the content with markers replaced by semantic HTML.
 */
function convertRefMarkersToHTML(
  content: string,
  researchItems: ResearchItem[]
): string {
  // Regex: /\{\{ref:(\d+)\}\}/g
  // For each match:
  //   1. Extract N (1-based index)
  //   2. Find researchItems[N-1]
  //   3. If found: replace with <span data-ref-id="UUID" data-ref-index="N" class="ref-indicator" contenteditable="false">N</span>
  //   4. If not found: strip the marker (graceful degradation)
}

/**
 * Strip all {{ref:N}} markers from content (for fallback/cleanup).
 */
function stripRefMarkers(content: string): string {
  return content.replace(/\{\{ref:\d+\}\}/g, '');
}
```

**Key decisions:**
- Markers use `{{ref:N}}` format — unlikely to conflict with normal content and easy to regex
- `contenteditable="false"` on the HTML span makes it an atomic unit in the editor
- Research items are resolved by 1-based index matching the order they were provided to the LLM

### 2. Content Writing Prompt Changes (`contentWritingTools.ts`)

**In `buildContentPrompt()`**, add after the Research Integration Rules section:

```typescript
const refMarkerInstruction = artifactType !== 'showcase' && researchContext ? `
### Reference Markers
After each sentence that was directly informed by, drawn from, or inspired by a specific research item, append a marker: {{ref:N}} where N is the research item number from the context above.

Rules:
- Only mark sentences where research DIRECTLY contributed (specific data, ideas, reasoning, examples)
- Do NOT mark every sentence — only those with clear research lineage
- A sentence can have multiple markers: "Sentence text.{{ref:2}}{{ref:5}}"
- Place markers AFTER terminal punctuation: "End of sentence.{{ref:3}}"
- If a sentence synthesizes from your own knowledge without research influence, do NOT add a marker
- Typical density: 30-50% of sentences should have markers (not every sentence, not too few)
` : '';
```

**In `writeContentSection()` and `writeFullContent()`**, after receiving LLM output and before returning:

```typescript
import { convertRefMarkersToHTML, stripRefMarkers } from './referenceMarkerUtils';

// After getting section content from LLM:
const contentWithRefs = convertRefMarkersToHTML(sectionContent, researchItemsForSection);
```

The `researchItemsForSection` is built from the research data already available in scope, with indices matching the order they were presented in the prompt's research context.

### 3. Humanizer Preservation (`humanityCheckTools.ts`)

**In `buildHumanityCheckPrompt()`**, add to the structural preservation section:

```typescript
const refMarkerPreservation = `
### Reference Markers (PRESERVE EXACTLY)
The content contains {{ref:N}} markers that link sentences to research sources.
These are structural metadata, NOT content. Rules:
- NEVER remove, modify, or reposition {{ref:N}} markers
- When rewriting a sentence, keep its {{ref:N}} marker(s) at the END of the rewritten sentence
```

### 4. TipTap ResearchRef Extension (`researchRef.ts`)

**Pattern to follow**: `frontend/src/lib/tiptap/inlineStyleCopy.ts` (existing custom extension)

```typescript
import { Node, mergeAttributes } from '@tiptap/core';

export interface ResearchRefOptions {
  HTMLAttributes: Record<string, any>;
}

declare module '@tiptap/core' {
  interface Commands<ReturnType> {
    researchRef: {
      // No commands needed — nodes are created from HTML parsing
    };
  }
}

export const ResearchRef = Node.create<ResearchRefOptions>({
  name: 'researchRef',
  group: 'inline',
  inline: true,
  atom: true,          // Treated as single unit, not partially selectable
  selectable: true,    // Can be selected and deleted
  draggable: false,

  addAttributes() {
    return {
      refId: {
        default: null,
        parseHTML: (el) => el.getAttribute('data-ref-id'),
        renderHTML: (attrs) => ({ 'data-ref-id': attrs.refId }),
      },
      refIndex: {
        default: null,
        parseHTML: (el) => el.getAttribute('data-ref-index'),
        renderHTML: (attrs) => ({ 'data-ref-index': attrs.refIndex }),
      },
    };
  },

  parseHTML() {
    return [{ tag: 'span.ref-indicator[data-ref-id]' }];
  },

  renderHTML({ HTMLAttributes }) {
    return [
      'span',
      mergeAttributes(this.options.HTMLAttributes, HTMLAttributes, {
        class: 'ref-indicator',
        contenteditable: 'false',
      }),
      HTMLAttributes.refIndex || '',
    ];
  },
});
```

## Data Model

### No Schema Changes Required

The `artifact_research` table already contains all fields needed:
- `id` (UUID) — referenced by `data-ref-id` in HTML
- `source_name` — displayed in hover card title
- `source_type` — determines icon
- `source_url` — "Open Source" button target (nullable)
- `excerpt` — first sentence shown in hover card

Reference mappings are stored **inline in HTML** as data attributes on `<span>` elements. No additional table or JSONB field needed. This is deliberate:
- Mappings are content — they live with the content
- Deleting an indicator = deleting the span from HTML = mapping gone
- No orphaned data or sync issues

## API Design

No new API endpoints. All data is served by existing endpoints:
- `GET /api/artifacts/:id` — returns artifact with content containing ref-indicator spans
- `GET /api/artifacts/:id/research` — returns research items (already used by `useResearch()`)

## Error Handling

| Error Scenario | Handling Strategy |
|----------------|-------------------|
| LLM outputs malformed markers (e.g. `{{ref:abc}}`) | Regex only matches `\d+`; malformed markers are stripped |
| Research item deleted after content generated | Indicator shows "Source unavailable" in hover card |
| `{{ref:N}}` where N exceeds research count | Marker silently stripped during post-processing |
| Humanizer removes markers despite instruction | Post-processing fallback: if zero markers survive, log warning but don't fail |
| HoverCard fails to load research data | Card shows skeleton/loading state, then "Unable to load source" |

- **Rollback plan**: If issues arise, remove the `{{ref:N}}` instruction from `buildContentPrompt()` — new artifacts generate without markers; existing indicators remain in HTML but are harmless
